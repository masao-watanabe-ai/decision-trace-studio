# Decision Trace Studio — デモ スピーカーノート

---

## デモの目的

このデモで伝えたいことは一つです。

> **「意思決定の構造を設計して、実行して、改善できる」**

AIが答えを出すのではなく、人間が決めた判断基準をフローとして設計し、そのフローをシミュレーションで検証して、継続的に品質を高めていく——その一連のサイクルをこの Studio で回せることを体験してもらいます。

コールセンターの問い合わせ対応を題材に、3分で完結するデモです。

---

## 30秒イントロ

> *(相手に向かって、スライドや画面を出す前に話す)*

「コールセンターで毎日何百件もの問い合わせを受けているとします。VIPのお客様には優先対応、法的リスクがある案件はエスカレーション、クレームは人間に回す——こういった判断基準は、現場の誰かの頭の中にあることが多いですよね。」

「Decision Trace Studio は、その判断基準をフローとして設計して、テストして、改善できる Studio です。AIに判断を任せるのではなく、人間が設計した判断の構造を可視化して、品質を継続的に高めていくためのツールです。実際に動かしながら3分でお見せします。」

---

## 3分デモ台本（全体の流れ）

| 時間 | セクション | 伝えること |
|---|---|---|
| 0:00〜0:30 | Design | 判断フローの構造を見せる |
| 0:30〜1:10 | Simulate + Trace | フローを実行して根拠を確認する |
| 1:10〜1:50 | Compare | 変更の影響を定量的に確認する |
| 1:50〜2:40 | Improve | 改善提案を承認してフローに反映する |
| 2:40〜3:00 | 締め | サイクルの価値をまとめる |

## 画面操作ごとのスピーカーノート

---

### 【Design】判断フローの構造を見せる（0:00〜0:30）

**操作:** Studio を開いた状態で Design タブが表示されている。  
ヘッダー右上の **`▶ Demo`** ボタンを押してガイドを起動する。  
Step 1 の状態でノードをクリックして NodeDetailPanel に条件・アクションを表示する。  
Boundary ノード（Refund+Legal Risk）を選択してハイライトする。

**話す内容:**

「これがコールセンター対応フローです。4つのノードが並んでいます。」

「左上の VIP Fast Track は、VIPフラグが立っているお客様を即座に優先対応するノードです。その下の Refund+Legal Risk は、一定金額以上の返金リクエストかつ法的な言及があるケースを捕まえる Boundary ノードです。」

「Boundary というのは、このラインを超えたら別の対応に切り替えるという境界線です。ここに引っかかったケースは自動対応ではなく、エスカレーションのアクションが実行されます。」

「右側のパネルに条件とアクションが表示されていますね。これが判断の構造です。頭の中にある判断基準が、ここに可視化されています。」

---

### 【Simulate】フローを実行して確認する（0:30〜1:10）

**操作:** Simulate タブに切り替える（Demo Step 2）。  
「シナリオ生成」ボタンをクリックする。  
生成されたシナリオ一覧を見せる。  
「全シナリオ実行」ボタンをクリックする。  
実行結果が出たら、任意のシナリオ行の **Trace** リンクをクリックして詳細ページを開く。

**話す内容:**

「Simulate タブに移ります。シナリオ生成を押すと、このフローに対するテストケースが自動で生成されます。VIPのお客様、法的リスクあり、クレーム対応、通常問い合わせ、といったパターンが出てきています。」

「全件実行を押すと、各シナリオがフローを走行して、どのノードを通過してどのアクションが選ばれたかが記録されます。」

「この Trace リンクを開くと——」

*(Trace ページに遷移)*

「このシナリオがノードごとにどう評価されたかのステップが時系列で確認できます。なぜこの判断になったのか、を後から追えるようになっています。これが説明責任の担保になります。」

---

### 【Compare】変更の影響を定量的に確認する（1:10〜1:50）

**操作:** Compare タブに切り替える（Demo Step 4）。  
Before / After のカラムが並んだ状態を見せる。  
*(このタイミングでは Before のみのデータ。続けて Improve 後に戻ってくることを予告する。)*  
「変更後にここが変わります」と伝えつつ画面構成を説明する。

**話す内容:**

「Compare タブです。フローを変更する前と後のシミュレーション結果をこのように並べて比較できます。」

「どのシナリオのアクションが変わったか、逆にどのシナリオは影響を受けなかったかが赤と緑のハイライトで一目でわかります。」

「フローを改善するとき、直感だけで変更を加えると予期しない副作用が出ることがあります。Compare を使えば、変更の影響範囲を本番反映前に定量的に把握できます。このあと Improve でフローを変更して、ここに戻ってきます。」

---

### 【Improve】提案を承認してフローに反映する（1:50〜2:40）

**操作:** Improve タブに切り替える（Demo Step 5）。  
「改善提案を生成」ボタンをクリックする。  
提案一覧が表示されたら内容を読み上げる。  
提案の「承認 & Flow 反映」ボタンをクリックする。  
Design タブに戻って、ノードが追加・変更されていることを確認する。  
再度 Simulate タブに戻って全件実行する（Demo Step 7）。  
Compare タブで差分を確認する。

**話す内容:**

「Improve タブです。改善提案を生成ボタンを押すと、シミュレーション結果を分析した提案が出てきます。」

「たとえばここに『Fallback に落ちているケースに対して Boundary を追加することで捌ける』という提案があります。提案の条件とアクションも自動で設定されています。」

「承認ボタンを押します。」

*(クリック)*

「Design タブに戻ると、フローにノードが追加されています。変更後のフローで再度シミュレーションを実行して——」

*(Simulate 全件実行)*

「Compare に戻ると、Before と After で変わったシナリオが色で示されています。今回の改善が意図した効果をもたらしているかが、数値で確認できました。」

---

### 【締め】サイクルの価値をまとめる（2:40〜3:00）

**操作:** Demo の「完了」ボタンを押してガイドを閉じる。  
Design タブを表示した状態で話す。

**話す内容:**

「設計して、実行して、比較して、改善する。このサイクルを一つの Studio の中で回せるのが、Decision Trace Studio の価値です。」

「大事なのは、AIに判断を委ねているのではなく、人間が設計した判断基準を可視化して、品質を継続的に高めていくということです。判断の根拠が残るので、後から監査もできますし、チームで判断基準を共有することもできます。」

「以上です。何かご質問はありますか？」

## 想定質問と短い回答

| 質問 | 回答 |
|---|---|
| これは AI が判断するツールですか？ | いいえ。AIはシミュレーションや提案生成の補助に使いますが、判断ロジックを設計するのは人間です。AIはシグナルを出す役割、判断基準を決めるのは人間という分担です。 |
| データは本番と繋がっていますか？ | 現在はデモ用のインメモリ実装です。本番導入時は業務システムの問い合わせデータをインプットにシミュレーションを実行できます。 |
| フローはどれくらい複雑にできますか？ | ノードとエッジで表現できる範囲であれば柔軟に設計できます。現在は Decision / Boundary / Human Gate / Fallback の4種類のノードをサポートしています。 |
| Compare は何回分比較できますか？ | 直近のシミュレーション結果を Before / After として比較します。より詳細なバージョン管理への拡張は今後のロードマップです。 |
| Trace はどのくらい残りますか？ | 現在はインメモリなので再起動でリセットされます。本番想定ではDBに永続化します。 |
| Human Gate はどう使いますか？ | 条件を満たしたケースを自動的に「人間にエスカレーション」として記録するノードです。アクションは `human_review` 固定で、自動判断を意図的にバイパスします。 |
| 既存のシステムと連携できますか？ | バックエンドは FastAPI + REST API 構成なので、既存システムからのシナリオ投入や判断結果の取得はAPIで対応できます。 |
